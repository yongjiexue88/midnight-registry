## 1. 执行摘要

当前项目已经从宠物小镇原型转向《午夜登记簿》，并形成了可运行的七夜门岗身份核验游戏。

核心审计结果：

- 视觉素材共 **77 个 PNG**：16 个角色、39 个道具、8 个监控场景、10 个准备台素材、4 张总览图。
- 音频素材共 **15 个 WAV**：电话、扫描、CCTV、印章、警报、数值受损、雨声和换班提示。
- 动画设计清单共 **56 个事件**，CSS 中有 **52 个独立关键帧**。两者不能直接一一对应，因为部分事件复用同一个动画，也有部分关键帧服务于清单外的通用反馈。
- 实施前 56 个动画事件中，审计判定为：
  - **已接入：38 项**
  - **部分接入：13 项**
  - **未形成有效运行态：5 项**
- 素材校验脚本通过：16 个角色为 `512 x 768`，39 个道具和 8 个监控场景为 `512 x 512`。
- TypeScript 类型检查通过。
- 当前实际可玩来访案例统一为 **68 个**；设计计划中多出的 3 个未实现条目已移除，避免 68/71 双重口径。
- 第三夜实体薛夜复制体已移除，改为错误换班单和前台监控单帧异常；第七夜保留为唯一实体终局。

## 4. 动画系统判断

### 4.1 已经做得较完整的部分

- 环境压力、来客情绪、文档切换、四种裁决和夜间结算已经形成统一视觉语言。
- 动画主要承担状态反馈，而不是单纯装饰。
- “留置等待”具有沙漏、进度、二次来源和重新裁决，是目前互动性最强的动画流程。
- 镜像来客具备重影、眼部故障、错误影子、暴露和现实改写等连贯恐怖反馈。

### 4.2 当前验证脚本的局限

`npm run verify:assets` 只验证：

- 文件数量。
- PNG 尺寸。
- `registryAnimationAssets` 中是否存在 56 条登记数据。

它不会验证：

- 56 个动画是否真的有组件触发。
- 触发条件是否符合设计描述。
- 动画是否能在玩家流程中到达。
- 是否存在只定义未使用的状态。

因此，“56 个动画事件验证通过”不能等同于“56 项全部实现”。

### 4.3 优先补齐的动画

第一优先级：

1. 来客说话状态。
2. 电话死寂和异常通话即时失真。
3. 扫描通过、扫描错误。
4. CCTV 放大定格。
5. 正确裁决印章。

第二优先级：

1. 文件详情打开和关闭。
2. 声誉、安全受损的独立反馈。
3. 日光灯和走廊影子改为受稳定度、夜数和威胁控制。
4. 所有复制体成功进入时统一播放登记簿改写。

## 5. 核心剧情

玩家薛夜是月影公寓新来的临时夜班门岗。工作表面上是核对住户和访客，真正的职责却是决定谁会被登记簿承认为“现实中的人”。

门外的冒名者不是普通怪物，而是登记系统中的错误记录。它们可以复制证件、档案和已经写下的事实，却难以复制没有进入记录的私人记忆、生活习惯、关系和问候方式。

一旦玩家盖下允许印章，登记簿就会把错误身份写成正式现实。复制体会占据真实住户的姓名和位置，原住户则可能被困在门外、从档案中消失，甚至被所有人遗忘。

玩家必须在四种处理之间选择：

- **允许进入**：把来客正式登记为可进入者。
- **拒绝进入**：阻止证据不符或违反规则的人。
- **呼叫安保**：处理复制声音、危险冒名者和最终身份覆盖。
- **留置等待**：争取回拨、监控变化和来客反应，但会增加时间、声誉和危险压力。

主线逐步揭示：

1. 公寓过去发生过一次整栋楼的登记错误。
2. 已注销十二年的蓝星维修仍在生成工单。
3. 物业和外部系统并不可靠。
4. 档案本身会被污染并改写。
5. 冒名者会学习玩家最常用的核验流程。
6. 薛夜自己的档案状态是“等待替换”。
7. 最后一名来客是拥有完整文件的第二个薛夜。

### 7.1 目标案例数与实际案例数不同

- 第二夜 10。
- 第三夜 9。
- 第四夜 11。
- 第五夜 12。
- 第六夜 13。

- 前三夜各 8 个。
- 后四夜分别为 11、12、13、8 个。

缺少的设计案例：

此外，`visitorsPerNight = 8` 与第四至第六夜的实际数量不一致，容易误导后续逻辑和文档。

### 7.2 第三夜提前使用了第七夜高潮

第三夜最后一案已经是：

- 薛夜复制体。
- 000 室。
- 没有换班安排。
- 监控中没有影子。
- 登记簿已有薛夜值班记录。
- 工牌背面写着“不得登记第二个自己”。
- 正确处理为呼叫安保。

第七夜最终案例再次使用几乎相同的结构和证据。

这会造成两个问题：

1. 玩家在第三夜就已经知道最终秘密，第七夜缺少新的认知反转。
2. 七夜中档案污染和学习型复制体的升级没有真正改变最终解法。

建议把第三夜案例改成“错误的换班通知”或“监控里短暂出现另一个前台”，只暴露部分线索，不让第二个薛夜正式站到门外。第七夜再完成实体来访和身份覆盖高潮。

### 7.3 第三夜设计决策不一致

设计计划中的第三夜薛夜案例正确答案是“拒绝”，当前手工数据中是“呼叫安保”。
考虑到复制声音、无影子和身份覆盖风险，“呼叫安保”更符合现有四级危险规则，但应统一设计文档。

### 7.4 监控素材利用率不足

项目有 8 个 CCTV 场景，但玩家监控终端只有 4 个频道。后 4 个场景与第五至第七夜剧情高度相关，未接入会降低后期夜晚的视觉升级感。

建议按夜晚解锁：

- 第五夜：污染档案室。
- 第六夜：不可能楼梯间、蓝星车辆。
- 第七夜：安保岗亭和前台自身监控。

## 8. 互动系统与剧情作用

已经接入的主要互动：

- 文件标签切换和住户档案比对。
- 今夜通知、预约和出入登记簿。
- 电话交换机，可联系房间、物业、安保和邻居。
- 四频道 CCTV。
- 证件扫描。
- 身份、关系、情境和陷阱四类问题。
- 六项人工核验清单和八类证据理由。
- 允许、拒绝、安保、留置四种裁决。
- 留置后的回拨、监控、观察反应三选一调查。
- 电话接线、监控调频、扫描校准、门锁密码等故障小游戏。
- 白日档案修复、设备支援和永久升级。
- 故事、停电挑战、无尽值班三种模式。
- 住户被替换、被困门外或保持有效的持续状态。
- 冒名者学习玩家最常使用工具的机制。

最能支撑故事主题的交互是：

1. 未记录记忆和私人习惯提问。
2. 电话中“本人已经在楼内”的矛盾。
3. 留置后证据发生变化。
4. 错误放行会改变住户持续状态。
5. 最终必须证明“当前薛夜已经被登记”，而不是单纯识别怪物外貌。

## 9. 结局

当前共有四种结局：

### 好结局

阻止最终复制体，没有放入复制体或错误很少，并保留足够高的总分。天亮时招聘告示上的名字被划掉，薛夜离开登记循环。

### 幸存结局

熬过七夜，但有太多真实住户被替换。公寓仍然运转，所有人的行为都与记忆略有偏差。

### 坏结局

允许前台复制体进入，或安全、声誉、稳定度降到零。清晨，新的薛夜坐在前台内侧，真正的薛夜站在雨中请求进入。

### 隐藏结局

最终案件中保持较高安全、声誉和稳定度，并标记至少三条证据，可选择烧毁登记簿。复制体和薛夜自己的公寓记录都会被删除。

仍保留英文的主要位置：

## 11. 整改优先级

### P0：剧情连续性

1. 重写第三夜最后一案，避免完整复用第七夜终局。

3. 明确项目采用 68 案例还是补齐到 71 案例。

### P1：缺失动画反馈

3. 增加正确裁决、声誉受损、安全受损的专属反馈。

### P3：数据维护

4. 修复 `evidenceOptions` 中重复的“未记录记忆错误”条目。
